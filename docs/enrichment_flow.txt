LLM-BASED THREAT INTELLIGENCE GATHERING SYSTEM: ENRICHMENT ARCHITECTURE
==========================================================================

The Enrichment layer is designed as a multi-stage pipeline that transitions from deterministic data extraction to complex semantic reasoning. This layer ensures that raw threat data is enriched with technical, behavioral, and strategic context before reaching the final analyst.

STAGE 1: DETERMINISTIC INDICATOR EXTRACTION
--------------------------------------------
- Primary File: entity_extractor.py
- Mechanism: Pre-compiled Regular Expressions (Regex).
- Purpose: Identifies "Hard IOCs" such as CVE IDs, IPv4/IPv6 addresses, Domains, and Cryptographic Hashes (MD5, SHA1, SHA256).
- Strategy: By performing this extraction first, the system handles high-volume, fixed-pattern data with 100% precision, ensuring that the LLM in later stages does not need to "guess" technical identifiers.

STAGE 2: SEMANTIC ENTITY RECOGNITION (NER)
------------------------------------------
- Primary File: ner_spacy.py
- Mechanism: spaCy NLP Pipeline with a custom 'EntityRuler'.
- Purpose: Extracts "Soft Entities" like Threat Actor groups (e.g., Lazarus, APT41) and Malware families (e.g., Ryuk, Emotet).
- Strategy: The base 'en_core_web_sm' model is augmented with a custom pattern catalogue of 60+ entries. This ensures that domain-specific jargon (like 'Fancy Bear') is correctly labeled as THREAT_ACTOR rather than a generic person or organization.

STAGE 3: THE KNOWLEDGE ENGINE - FEW-SHOT MAPPING
------------------------------------------------- 
- Primary Files: few_shot_examples.py (Knowledge Base) & attack_mapper.py (Execution Engine).
- Mechanism: In-Context Learning (ICL) using Llama 3 via LangChain.
- Deep Dive - few_shot_examples.py:
    * Purpose: This file acts as the "Brain" of the mapper. LLMs often struggle with specific output formats (JSON) or complex frameworks like MITRE ATT&CK without guidance.
    * Contents: It contains 5 curated, high-fidelity examples:
        1. Initial Access: Mapping Log4Shell to T1190.
        2. Execution/Lateral Movement: Mapping PowerShell/SMB to T1059.001 and T1021.002.
        3. Exfiltration: Mapping encrypted ZIPs and Dropbox API to T1560.001 and T1048.002.
        4. Persistence: Mapping WannaCry Registry keys to T1053.005 and T1547.001.
        5. Command & Control: Mapping TLS 1.3 and Domain Fronting to T1071.001 and T1090.004.
    * Logic: It provides the 'SYSTEM_PREFIX' (defining the analyst persona) and 'EXAMPLE_PROMPT' (structuring the input/output pair) which are essential for stable LLM performance.
- Deep Dive - attack_mapper.py:
    * Logic: It combines the few-shot examples with the 'cleaned_text' into a single prompt.
    * Anti-Hallucination: After the LLM generates TTP IDs, this script validates them against 'enterprise-attack.json'. If an ID (e.g., T9999) does not exist in the official MITRE framework, it is discarded.

STAGE 4: ANALYST SYNTHESIS & RAG REPORTING
------------------------------------------
- Primary File: report_generator.py
- Mechanism: Closed-Domain Retrieval-Augmented Generation (RAG).
- Purpose: Produces the final executive summary.
- Strategy: The LLM is strictly constrained to the 'Context' (Cleaned Text + Extracted Entities + Verified TTPs). It is mandated to include [source_id: X] citations for every claim. If the data is insufficient, it must return a standard "Insufficient data to determine" response to prevent analyst misinformation.
